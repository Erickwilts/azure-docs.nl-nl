#### <a name="service-tags"></a>Servicetags
Servicetags vereenvoudigen de beveiliging voor virtuele Azure-machines en Azure-virtuele netwerken door dat u de netwerktoegang tot de Azure-services eenvoudig beperken. U servicetags gebruiken in de NSG-regels (Network Security Group) om verkeer toe te staan of te weigeren naar een specifieke Azure-service wereldwijd of per Azure-regio. Azure biedt het onderhoud van IP-adressen die aan elke tag ten grondslag liggen. HDInsight-servicetags voor netwerkbeveiligingsgroepen (NSG's) zijn groepen IP-adressen voor gezondheids- en beheerservices. Deze groepen helpen de complexiteit voor het maken van beveiligingsregels te minimaliseren. HDInsight-klanten kunnen servicetag inschakelen via Azure Portal, PowerShell en REST API. Zie [NSG-servicetags (Network security group) voor Azure HDInsight voor](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags)meer informatie.

#### <a name="custom-ambari-db"></a>Aangepaste Ambari-database
Met HDInsight u nu uw eigen SQL DB gebruiken voor Apache Ambari. U deze aangepaste Ambari DB configureren vanuit de Azure-portal of via de sjabloon resourcemanager.  Met deze functie u de juiste SQL DB kiezen voor uw verwerkings- en capaciteitsbehoeften. U ook eenvoudig upgraden om aan de vereisten voor bedrijfsgroei te voldoen. Zie [HDInsight-clusters instellen met een aangepaste Ambari DB](hdinsight-custom-ambari-db.md)voor meer informatie.

#### <a name="transport-layer-security-tls-12-enforcement"></a>Transport Layer Security (TLS) 1.2 handhaving
Tls (Transport Layer Security) en Secure Sockets Layer (SSL) zijn cryptografische protocollen die communicatiebeveiliging bieden via een computernetwerk. Zie [Transport Layer Security](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0)voor meer informatie. Hoewel Azure HDInsight-clusters TLS 1.2-verbindingen accepteren op openbare HTTPS-eindpunten, wordt TLS 1.1 nog steeds ondersteund voor achterwaartse compatibiliteit met oudere clients.

Vanaf de volgende release u zich aanmelden en uw nieuwe HDInsight-clusters configureren om alleen TLS 1.2-verbindingen te accepteren. 

Later in het jaar, vanaf 6/30/2020, zal Azure HDInsight TLS 1.2 of hoger versies afdwingen voor alle HTTPS-verbindingen. We raden u aan ervoor te zorgen dat al uw klanten klaar zijn om TLS 1.2 of latere versies te verwerken.

## <a name="release-date-11072019"></a>Releasedatum: 11/07/2019

Deze release geldt zowel voor HDInsight 3.6 als 4.0.

### <a name="new-features"></a>Nieuwe functies

#### <a name="hdinsight-identity-broker-hib-preview"></a>HDInsight Identity Broker (HIB) (Preview)

HDInsight Identity Broker (HIB) stelt gebruikers in staat om zich aan te melden bij Apache Ambari met behulp van multi-factor authenticatie (MFA) en de vereiste Kerberos-tickets te krijgen zonder wachtwoordhashes in Azure Active Directory Domain Services (AAD-DS) te hoeven verkrijgen. Momenteel is HIB alleen beschikbaar voor clusters die zijn geïmplementeerd via de ARM-sjabloon (Azure Resource Management).

#### <a name="kafka-rest-api-proxy-preview"></a>Kafka Rest API Proxy (Preview)

Kafka Rest API Proxy biedt one-click implementatie van zeer beschikbare REST proxy met Kafka cluster via beveiligde AAD autorisatie en OAuth protocol. 

#### <a name="auto-scale"></a>Automatisch schalen

Autoscale voor Azure HDInsight is nu algemeen beschikbaar in alle regio's voor Apache Spark- en Hadoop-clustertypen. Deze functie maakt het mogelijk om big data analytics workloads op een meer kostenefficiënte en productieve manier te beheren. Nu u het gebruik van uw HDInsight-clusters optimaliseren en alleen betalen voor wat u nodig hebt.

Afhankelijk van uw vereisten u kiezen tussen op basis van laden en op planning gebaseerde autoscaling. Op basis van laden kan de clustergrootte op en neer schalen op basis van de huidige resourcebehoeften, terwijl op planning gebaseerde autoscale de clustergrootte kan wijzigen op basis van een vooraf gedefinieerde planning. 

Ondersteuning voor Autoscale voor HBase- en LLAP-workload is ook een openbare preview. Zie [Azure HDInsight-clusters automatisch schalen](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters)voor meer informatie.

#### <a name="hdinsight-accelerated-writes-for-apache-hbase"></a>HDInsight Accelerated schrijft voor Apache HBase 

Voor verbeterde schrijfbewerkingen worden beheerde Azure Premium SSD-schijven gebruikt om de prestaties van het Apache HBase Write Ahead Log (WAL) te verbeteren. Zie [Verbeterde schrijfbewerkingen van Azure HDInsight voor Apache HBase](https://docs.microsoft.com/azure/hdinsight/hbase/apache-hbase-accelerated-writes) voor meer informatie.

#### <a name="custom-ambari-db"></a>Aangepaste Ambari-database

HDInsight biedt nu een nieuwe capaciteit om klanten in staat te stellen hun eigen SQL DB voor Ambari te gebruiken. Nu kunnen klanten de juiste SQL DB voor Ambari kiezen en deze eenvoudig upgraden op basis van hun eigen groeivereiste. De implementatie gebeurt met een Azure Resource Manager-sjabloon. Zie [HDInsight-clusters instellen met een aangepaste Ambari DB](https://docs.microsoft.com/azure/hdinsight/hdinsight-custom-ambari-db)voor meer informatie.

#### <a name="f-series-virtual-machines-are-now-available-with-hdinsight"></a>Virtuele f-serie machines zijn nu beschikbaar met HDInsight

Virtuele F-serie virtuele machines (VM's) zijn een goede keuze om aan de slag te gaan met HDInsight met vereisten voor lichtverwerking. Tegen een lagere prijs per uur levert de F-serie de beste prijs/kwaliteit-verhouding van de Azure-portfolio, gebaseerd op de ACU (Azure Compute Unit) per vCPU. Zie [De juiste VM-grootte selecteren voor uw Azure HDInsight-cluster voor](https://docs.microsoft.com/azure/hdinsight/hdinsight-selecting-vm-size)meer informatie.

### <a name="known-issues"></a>Bekende problemen

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>Replicatie is verbroken voor Secure HBase met standaardinstallatie

Ga voor HDInsight 4.0 de volgende stappen uit:

1. Interclustercommunicatie inschakelen.
1. Meld u aan bij de actieve headnode.
1. Download een script om replicatie in te schakelen met de volgende opdracht:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Typ de opdracht `sudo kinit <domainuser>`.
1. Typ de volgende opdracht om het script uit te voeren:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
Ga voor HDInsight 3.6 als volgt te werk:

1. Meld u aan bij actieve HMaster ZK.
1. Download een script om replicatie in te schakelen met de volgende opdracht:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Typ de opdracht `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>`.
1. Typ de volgende opdracht:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

#### <a name="phoenix-sqlline-stops-working-after-migrating-hbase-cluster-to-hdinsight-40"></a>Phoenix Sqlline werkt niet meer na migratie van HBase-cluster naar HDInsight 4.0

Ga als volgt te werk:

1. Laat de volgende Phoenix-tabellen vallen:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.MUTEX`
    1. `SYSTEM.CATALOG`
1. Als u geen van de tabellen verwijderen, start u HBase opnieuw op om alle verbindingen met de tabellen te wissen.
1. Voer `sqlline.py` opnieuw uit. Phoenix zal opnieuw alle tabellen die zijn verwijderd in stap 1 opnieuw maken.
1. Regenereren Phoenix tabellen en weergaven voor uw HBase gegevens.

#### <a name="phoenix-sqlline-stops-working-after-replicating-hbase-phoenix-metadata-from-hdinsight-36-to-40"></a>Phoenix Sqlline werkt niet meer na repliceren HBase Phoenix metadata van HDInsight 3.6 naar 4.0

Ga als volgt te werk:

1. Ga voordat u de replicatie uitvoert naar het `sqlline.py`cluster bestemming 4.0 en voert u uit. Deze opdracht genereert Phoenix `SYSTEM.MUTEX` `SYSTEM.LOG` tabellen zoals en die alleen bestaan in 4.0.
1. De volgende tabellen laten vallen:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.CATALOG`
1. De HBase-replicatie starten

#### <a name="mahout"></a>Mahout

In HDP-2.3.x en 2.4.x hebben we in plaats van een specifieke Apache-release van Mahout te verzenden, gesynchroniseerd met een bepaald revisiepunt op Apache Mahout trunk. Dit revisiepunt is na de 0.9.0-release, maar vóór de 0.10.0-release. Dit biedt een groot aantal bugfixes en functionele verbeteringen ten opzichte van de 0.9.0-release, maar biedt een stabiele release van de Mahout-functionaliteit vóór de volledige conversie naar nieuwe Spark-gebaseerde Mahout in 0.10.0.

Het revisiepunt dat voor Mahout in HDP 2.3.x en 2.4.x is gekozen, is van de "mahout-0.10.x" tak van Apache Mahout, met 19 december 2014, revisie 0f037cb03e77c096 in GitHub.

In HDP-2.5.x en 2.6.x hebben we de "commons-httpclient"-bibliotheek van Mahout verwijderd omdat we deze zien als een verouderde bibliotheek met mogelijke beveiligingsproblemen en de Hadoop-Client in Mahout hebben geüpgraded naar versie 2.7.3, dezelfde versie die wordt gebruikt in HDP-2.5. Als gevolg hiervan:

-   Eerder gecompileerde Mahout-taken moeten opnieuw worden gecompileerd in de HDP-2.5- of 2.6-omgeving.

-   Er is een kleine kans dat sommige Mahout-taken "ClassNotFoundException" of "kon niet laden klasse" fouten met betrekking tot "org.apache.commons.httpclient", "net.java.dev.jets3t", of gerelateerde klasse naam prefixes tegenkomen. Als deze fouten zich voordoen, u overwegen om handmatig de benodigde potten in uw klassenpad voor de taak te installeren, als het risico van beveiligingsproblemen in de verouderde bibliotheek acceptabel is in uw omgeving.

-   Er is een nog kleinere kans dat sommige Mahout-taken crashes kunnen tegenkomen in mahout's hbase-clientcodeoproepen naar de veelvoorkomende bibliotheken van Hadoop, als gevolg van binaire compatibiliteitsproblemen. Helaas is er geen manier om dit probleem op te lossen, behalve terug te keren naar de HDP-2.4.2 versie van Mahout, die beveiligingsproblemen kan hebben. Nogmaals, dit moet zeer ongebruikelijk zijn, en is het onwaarschijnlijk dat zich voordoen in een bepaalde Mahout job suite.

      > [!NOTE]  
      > Alleen wanneer Ranger is ingeschakeld. 
     
  Dit probleem doet zich voor wanneer u probeert te upgraden naar HDP-2.6.1 ten opzichte van eerdere HDP-versies via Ambari. Ambari maakt gebruik van een krul oproep te doen een service check naar Ranger service in Ambari. Als de JDK-versie die door Ambari wordt gebruikt JDK-1.7 is, mislukt het curlinggesprek met de onderstaande fout:
    
  `curl: (35) error:14077410:SSL routines:SSL23_GET_SERVER_HELLO:sslv3 alert handshake failure`
    
  De reden voor deze fout is de tomcat versie die wordt\*gebruikt in Ranger is Tomcat-7.0.7 . Het gebruik van JDK-1.7 is in strijd met standaardcijfers\*in Tomcat-7.0.7 .
    
  U dit probleem op twee manieren oplossen:
    
  -   Werk de JDK die in Ambari wordt gebruikt bij van JDK-1.7 naar JDK-1.8 (zie de sectie [De JDK-versie wijzigen](https://docs.hortonworks.com/HDPDocuments/Ambari-2.2.0.0/bk_ambari_reference_guide/content/ch_changing_the_jdk_version_on_an_existing_cluster.html) in de Ambari-referentiegids).
    
  -   Als u een JDK-1.7-omgeving wilt blijven ondersteunen:
    
      1.  Voeg de property ranger.tomcat.ciphers toe in de ranger-admin-site sectie in uw Ambari Ranger configuratie met de onderstaande waarde:
            
          SSL\_\_RSA\_MET\_RC4\_128\_MD5, SSL RSA\_MET\_RC4\_128\_\_SHA, TLS\_\_\_\_\_\_\_RSA\_MET AES\_128 CBC\_SHA, SSL RSA MET 3DES EDE CBC\_SHA
        
      2.  Als uw omgeving is geconfigureerd voor Ranger-KMS, voegt u de property ranger.tomcat.ciphers toe in het gedeelte ranger-kms-site in uw Ambari Ranger-configuratie met de onderstaande waarde:
            
          SSL\_\_RSA\_MET\_RC4\_128\_MD5, SSL RSA\_MET\_RC4\_128\_\_SHA, TLS\_\_\_\_\_\_\_RSA\_MET AES\_128 CBC\_SHA, SSL RSA MET 3DES EDE CBC\_SHA
    
  >[!NOTE]  
  >De bekende waarden zijn werkvoorbeelden en zijn mogelijk niet indicatief voor uw omgeving. Zorg ervoor dat de manier waarop u deze eigenschappen instelt overeenkomt met de manier waarop uw omgeving is geconfigureerd.

Als u de nieuwste versie van Spark, Kafka en R Server (Machine Learning Services) wilt krijgen, kiest u de Spark-, Kafka-, ML Services-versie wanneer u [een HDInsight 3.6-cluster maakt.](https://docs.microsoft.com/azure/hdinsight/hdinsight-hadoop-provision-linux-clusters) Als u ondersteuning voor ADLS wilt krijgen, u het ADLS-opslagtype als optie kiezen. Bestaande clusters worden niet automatisch geüpgraded naar deze versies.

Alle nieuwe clusters die na juni 2018 zijn gemaakt, krijgen automatisch de meer dan 1000 bugfixes voor alle open-source projecten. Volg [deze](https://docs.microsoft.com/azure/hdinsight/hdinsight-upgrade-cluster) handleiding voor best practices rond upgraden naar een nieuwere HDInsight-versie.
